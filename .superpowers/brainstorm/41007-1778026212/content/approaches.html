Three Modeling Approaches

You said: downstream quality matters, clamping is good, GPU forward pass is OK

A

Classification Transformer ★ Recommended

Replace regression heads with classification. Block size → 64-class cross-entropy. Stab steps → ordinal regression (ordered 0-43). Shares one Transformer encoder. Captures multimodal distributions naturally.

Pros

  • Handles trimodal block_size directly
  • Ordinal loss respects 0 < 1 < 2 < ... without assuming equal spacing
  • Simple architecture change

Cons

  • 64 output classes may be sparse for tail values
  • No explicit boundary-awareness
B

Two-Stage Boundary-Aware Model

Stage 1: Binary classifier (boundary marker block_size=1 vs content block). Stage 2: Content blocks get bucketed classification (2-64). Stab steps predicted conditioned on block size bucket.

Pros

  • Explicitly models the 1→16→... alternation
  • Boundary detection is high-leverage (100% stab_step=0)

Cons

  • Error propagates from stage 1 to stage 2
  • More complex training loop
C

Bucketed + Ordinal (Simplest Entry)

Reduce output space. Bucket block_size into 9 natural groups: [1], [2-4], [5-8], [9-12], [13-15], [16], [17-20], [21-32], [33-64]. Treat stab_steps as ordinal. Same encoder, smaller heads.

Pros

  • Aligns with clamping consumption
  • 9 classes = less data-hungry
  • Tiny model change

Cons

  • Bucket boundaries are arbitrary
  • Cannot predict exact values if needed later

My Recommendation: Approach A

Why: the data distribution (peaks at 1, 16, geometric tail) is what classification was designed for. Ordinal loss for stab_steps naturally handles "mostly 0-6, occasionally up to 43." Adding bucketing (like treating block_size=1 as its own class, 16 as its own class, and grouping the tail) is a refinement we can add if 64-way classification underperforms for rare sizes.
