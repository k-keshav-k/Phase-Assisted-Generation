\papertitle{Phase-Adaptive Generation: Difficulty-Aware Semi-Autoregressive Decoding for Diffusion Language Models}

\paperauthor{Keshav Krishna \quad Prajesh Kumar \quad Siva Satvik Mandapati}
\paperaffiliation{New York University \\ Building LLM Reasoners --- Spring 2026 \\ \texttt{\{kk6081, pg2973, sm12779\}@nyu.edu}}

\begin{abstract}
Phase-Adaptive Generation (PAG) is an original research project on difficulty-aware decoding for diffusion language models. We first built a baseline pipeline around LLaDA and Dream traces: collect token-level confidence signals (top-1 probability, top-2 probability, entropy, $\tau_{\text{stability}}$, and $\tau_{\text{commit}}$), run change-point detection (CPD), derive segment tuples, and train a next-segment predictor. Through that process, we observed that CPD is useful for analysis but adds an avoidable intermediate step when the training target is block-level controls. Based on these observations, we now prioritize an AdaBlock-centered data path where traces directly provide block size and stabilization-step signals per block. This writeup documents both the attempted pipeline and the revised design decision, together with the current Transformer-based phase predictor and planned model ablations.
\end{abstract}

\section{Introduction}

Autoregressive (AR) language models generate one token at a time and therefore pay a latency cost that becomes significant on long reasoning traces. Diffusion-based LLMs (dLLMs) such as Dream 7B~\citep{ye2025dream} and LLaDA~\citep{nie2025llada} reduce this cost by generating in blocks through iterative denoising, but existing decoding policies are typically uniform: the same chunk size and refinement budget are applied regardless of whether the model is in an easy completion region or a difficult reasoning region.

Our project studies whether these generation traces contain reusable structure that can support better scheduling. The working hypothesis is that dLLM outputs move through difficulty phases: some spans stabilize quickly, while others remain uncertain and require more refinement. If those phases can be detected early enough, a scheduler can allocate compute more intelligently. In our current formulation, PAG aims to use phase predictions to adapt both chunk size and refinement intensity.

The writeup and codebase are organized to support that question in stages. We first collect traces from a baseline decoder, then segment the traces with change-point detection, then turn each segment into a tuple sequence, and finally predict the next segment tuple from previous ones. This staged setup is intentionally testable so that each step can yield intermediate evidence even before a full adaptive decoder is completed.

\section{Related Work}

\textbf{Diffusion LLM decoding.} Dream 7B~\citep{ye2025dream} and LLaDA~\citep{nie2025llada} show that diffusion-style decoding can be competitive with AR baselines on reasoning and planning tasks. Block Diffusion~\citep{arriola2025block} frames semi-autoregressive generation as a bridge between AR and diffusion decoding.

\textbf{Adaptive block inference.} AdaBlock-dLLM~\citep{lu2025adablock} is the closest prior idea to our project because it adapts block size based on confidence-like signals. Our project differs in two ways. First, we treat segmentation as an explicit learning problem over trace signals rather than only as a reactive confidence heuristic. Second, we predict a richer control tuple that can later drive both block size and refinement intensity.

\textbf{Why this formulation matters.} In the repository, the phase-analysis stage already exposes token-level traces, stabilizing features, and segment summaries. That makes the project a natural fit for a learned next-segment model: the model can operate on symbolic segment tuples rather than raw text, which keeps the method lightweight and interpretable.

\section{Method}

\subsection{Pipeline Overview}

The initial pipeline we implemented in the repository was:
baseline decoding $\rightarrow$ trace collection $\rightarrow$ CPD phase segmentation $\rightarrow$ tuple construction $\rightarrow$ next-segment prediction $\rightarrow$ tuple-conditioned unmasking.

After running this workflow on Dream 7B and LLaDA traces, we revised the design direction. Our current preferred pipeline is:
AdaBlock trace collection (block-native) $\rightarrow$ block tuple sequence construction $\rightarrow$ next-segment prediction $\rightarrow$ tuple-conditioned unmasking.

The key difference is that CPD is no longer required to recover block boundaries: AdaBlock traces already expose the block-level controls needed for training (block size and stabilization/refinement behavior), which reduces label-construction ambiguity and simplifies end-to-end training.

\subsection{Step 1: Baseline Trace Collection}

In the baseline trace-collection stage, we instrumented LLaDA and Dream generation to capture token-level trajectories over refinement steps. The tracked signals include top-1 probability, top-2 probability, entropy, and stabilization-oriented timing signals ($\tau_{\text{stability}}$, $\tau_{\text{commit}}$). These trajectories were used as raw material for downstream phase analysis.

This design gave us rich diagnostics but also revealed a bottleneck: the model controls we ultimately need for scheduling are block-level values, while LLaDA/Dream traces are token-level. As a result, we had to add a CPD stage to infer breakpoints before producing block-size labels. The revised plan is to use AdaBlock traces directly, where block size and stabilization-step information are already present at the block granularity needed for training.

\subsection{Step 2: Change-Point Detection and Segment Extraction}

The second stage of the initial pipeline applies CPD to stabilization-oriented feature series. We used PELT with configurable cost (L2/normal), penalty, minimum segment length, and smoothing-window controls, and experimented across Dream 7B and LLaDA traces using scalar signals such as stabilizing entropy, margin, and top-1 probability.

Qualitatively, PELT produced useful structure: some prompts show clear regime shifts that align with intuitive reasoning transitions, while others are sensitive to smoothing and penalty settings. This made CPD valuable for understanding model behavior but less attractive as a permanent label-generation dependency for predictor training.

Because our predictor target is block-level scheduling, we ultimately decided to use AdaBlock traces for training labels, where block size and stabilization-step signals are directly available without running CPD to recover breakpoints.

\subsection{Step 3: Tuple Sequences and Next-Segment Prediction}

The third stage represents each block/segment as an ordered control tuple and learns next-tuple prediction from history. In the current implementation (\texttt{phase\_predict}), a tuple contains at least $(\text{block\_size}, \text{refinement\_steps})$, with schema support for adding additional control dimensions later.

Our current predictor is a compact Transformer encoder trained as a regression model over normalized tuple sequences. Given a context window, the model predicts the next tuple and maps outputs back to integer scheduler controls. Compared with fixed-order heuristics, this architecture can model longer-range dependencies across blocks while remaining lightweight.

We treat this as a strong baseline rather than a final choice. Planned ablations include alternative sequence models (for example MLP/linear baselines, recurrent variants, and other Transformer sizes) to identify the best quality-latency trade-off for control prediction.

\subsection{Step 4: Tuple-Conditioned Diffusion Unmasking}

The fourth stage consumes the predicted tuple as a scheduling command for the next unmasking action. If the predicted tuple is $(b, \tau)$, block size $b$ controls how many positions are unmasked in the next decode block, and $\tau$ controls stabilization/refinement intensity before committing.

In the revised pipeline, this loop is designed around AdaBlock-native traces and controls: train on observed block sequences, predict the next block tuple, apply tuple-conditioned unmasking, append the realized block outcome, and repeat. This keeps the training target and inference control space aligned and removes the CPD-derived boundary recovery step from the production training path.

\section{Implementation Status}

The repository currently includes both the analysis-first pipeline and the revised training direction. The \texttt{phase\_cpd} package supports trace exploration, stabilization-feature extraction, and CPD experiments (including PELT controls and feature choices). The \texttt{phase\_predict} package implements tuple extraction, dataset construction, a Transformer next-segment predictor, training/inference wrappers, and checkpointing.

Practically, we view the CPD path as an exploratory analysis tool that helped us interpret phase structure in Dream/LLaDA traces, while the AdaBlock path is the intended data source for final predictor training because it provides block-size and stabilization-step signals directly. Remaining work is to complete the AdaBlock-integrated end-to-end loop and compare predictor variants under a common evaluation protocol.

\section{Preliminary Analysis and Limitations}

Our preliminary takeaway from CPD experiments is directional: PELT is effective for surfacing phase-like structure and for diagnosing where confidence dynamics shift, but it also introduces hyperparameter-sensitive boundary decisions when used as a mandatory label-construction step. That observation motivated the design change toward AdaBlock-native traces for predictor training.

The main current limitation is evaluation depth: we still need systematic held-out comparisons across predictor families and direct downstream measurements of quality, latency, and compute efficiency. A second limitation is that the present tuple schema is intentionally compact (block size and refinement/stabilization behavior), so richer controls may be needed if future experiments show underfitting of complex phase transitions.

\section{Conclusion}

PAG frames adaptive diffusion decoding as a next-control prediction problem over block/segment tuples. Our project progress includes an end-to-end attempt built on LLaDA/Dream traces plus PELT-based segmentation, and a revised pipeline decision informed by those results: train on AdaBlock-native traces where block size and stabilization-step controls are directly available. This revision simplifies label construction, aligns training and inference controls, and preserves CPD as a valuable analysis tool rather than a required production dependency.
